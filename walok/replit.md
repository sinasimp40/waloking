# NEXTREME GAMING HUB — Multi-Customer Launcher with OTA Updates

A premium futuristic game launcher (Electron + React + Vite) with companion server. Supports per-customer branded builds and over-the-air (OTA) updates from a self-hosted update server.

## Two Distinct Build Flows

### A. Per-Customer Branded Builds (recommended)
Build a separate branded folder for each customer (cafe). Each customer's build is wired to its own update channel and brand identity.

```cmd
build-customer.bat example-cafe     :: build one customer
build-all.bat                       :: build every customer in customers\
publish-update.bat --all            :: publish all built versions to update server
```

Each customer is one JSON file in `customers/`:
```json
{
  "channel": "example-cafe",
  "brandName": "EXAMPLE CAFE",
  "subtitle": "Premium Gaming Lounge",
  "logo": "branding/logo.png",
  "updateServer": "http://YOUR-RDP-IP:4231"
}
```

See `customers/README.md` for details.

### B. Legacy Single-Brand Build
The original `build.bat` (rebrand + build) still works but produces portable folder + zip outputs (no installer .exe), and the post-build verification steps (`collect-builds.js`, `verify-signatures.js`) may report warnings since they were written for `.exe` installers.

## OTA Update System

### Architecture
1. Each branded launcher is built with a baked-in `ota-config.json` (channel, version, update server URL).
2. Once installed, the launcher does TWO things in parallel:
   - **Live SSE push** (`electron/ota-live.js`) — opens a long-lived SSE connection to `<updateServer>/api/live/<channel>/<role>/<instance>`. The instant the admin clicks BUILD or UPLOAD UPDATE, the server pushes an `update` event and the launcher checks immediately. If SSE keeps failing (e.g. corporate proxy), the same module falls back to a 60s HTTP heartbeat that keeps the install registered as "online" in the admin panel.
   - **HTTP poll** (every 2 min) of `/updates/<channel>/latest.json` — kept as a belt-and-braces fallback for installs that briefly lose the live channel.
3. If a newer version exists, the launcher downloads the payload zip, verifies SHA-256, stages it in `.ota-pending/`, shows a cyberpunk progress modal, and prompts to restart.
4. On next launch, the staged zip is extracted over the install folder BEFORE the React app loads, then deleted.
5. Both the launcher and the server.exe show their running version in their window UI (titlebar version pill in the launcher, titlebar + footer in the server dashboard).

### Distribution Format
- `electron-builder` target is `["zip", "dir"]` with `asar: false`.
- Customer downloads `Nextreme-Gaming-Hub-Portable-<version>.zip`, extracts to any folder (e.g., `C:\Launcher\`), and runs `Launcher.exe`. **No installer wizard, no admin rights, no UAC.**
- OTA only updates the changed files inside that folder — small efficient updates.

### Update Server (`update-server/`)
A small Express server hosted on YOUR RDP / local PC, port **4231**.
- `start.bat` — launches the server AND adds Windows Firewall rule automatically (run as Admin first time)
- `server.js` — Express. Serves `/updates/<channel>/latest.json` + zip payloads (public) AND a full admin panel at `/admin/` (auth-required). Also exposes the live SSE registry (`/api/live/<channel>/<role>/<instance>`), HTTP heartbeat fallback, the admin online-stream (`/api/admin/online/stream`), the per-customer `/api/admin/customers/<channel>/upload-update` route, and the top-level multi-target `/api/admin/upload-update` route (target = single channel OR `__all__`) which runs as a tracked job and streams progress through the standard `/api/admin/jobs/<id>/stream` SSE pipeline.
- `db.js` — SQLite (better-sqlite3) at `update-server/data/launcher.db` (git-ignored). Customers table is the source of truth. On startup the server migrates any pre-existing `customers/*.json` files into the DB AND moves the originals to `customers/_migrated_backup/` (also git-ignored) as proof-of-import. After migration the live `customers/*.json` files are regenerated by a one-way DB→JSON mirror on every CRUD; this exists ONLY so the legacy `scripts/build-customer.js` and `scripts/build-all.js` keep working unchanged. Operators must NOT hand-edit those files — the next CRUD will overwrite them.
- `live.js` — in-memory online registry keyed by `channel + role + instance`, 90s eviction, fan-out broadcast helper used by both BUILD and UPLOAD-UPDATE.
- `cleanup.js` — `cleanupAfterBuild()` keeps only the newest version per channel under `releases/<channel>/`, `update-server/public/updates/<channel>/`, and the `-server` mirrors. Wired into the BUILD chain (post-publish step) and the UPLOAD-UPDATE handler.
- `public/admin/` — cyberpunk admin UI. Single per-customer **BUILD** button (always builds + ships + cleans up + pushes live), single global **BUILD ALL CUSTOMERS** button, top-level **UPLOAD PRE-BUILT UPDATE** form with a target dropdown (single channel or ALL CUSTOMERS) that streams progress through the BUILD CONSOLE, plus a per-customer **UPLOAD UPDATE** modal, live online/offline pill per customer with per-instance role + version + short instance id (driven by the admin SSE stream — no polling). Each customer card surfaces `[download]` links beside the published launcher/server version pointing at the latest payload zip on the OTA server. Topbar shows project version AND server version.
- **Per-customer auto-bump** — every BUILD click now resolves the next version from each customer's own DB row (`max(launcherVersion, serverVersion) + patch`), seeded from the global `package.json` for fresh customers. Server.js computes a `perChannelVersion` map BEFORE spawning, then injects `BUILD_VERSION` per spawned step into `scripts/build-customer.js` and `scripts/publish-update.js` (both honor the env var as an override of `package.json`). This means renaming a brand and clicking BUILD on the same channel always ships a strictly-greater version that the running launcher will accept — no more "I rebuilt but installs ignored it" confusion.
- `stop-firewall-rule.bat` — removes the firewall rule
- `README.md` — full setup + ops guide

Manual firewall command:
```cmd
netsh advfirewall firewall add rule name="NEXTREME-OTA-4231" dir=in action=allow protocol=TCP localport=4231
```

### Admin Panel Security
- Cookie-based HMAC-signed session (12h expiry, `httpOnly`, `sameSite=lax`)
- Password is read from `OTA_ADMIN_PASSWORD` env var only — set it manually (e.g. in `.env` or on the start.bat command line); the server refuses to boot without it. There is no auto-generated `.admin-password` file anymore.
- 5 failed login attempts per IP → 5-minute lockout
- Strict channel name validation (`^[a-z0-9][a-z0-9-]{0,49}$`) on every route to prevent path traversal in logo uploads and build/publish args
- `child_process.spawn` uses `shell:false` with fixed script paths and validated args; wrapped in try/catch so a malformed payload can't crash the server
- SSE log stream uses cookie auth only (no token-in-URL leakage)
- `scripts/build-customer.js` invokes `rebrand.js` via `execFileSync` (argv array, no shell) so customer-controlled `brandName` / `subtitle` strings can never inject shell metacharacters

### Build Pipeline Resilience (admin BUILD)
The admin BUILD route runs as a **chained job** (sequence of steps with abort-on-fail):
1. `npm install` in project root — **skipped** if `node_modules/.bin/vite` and `electron-builder` already exist
2. `npm install` in `server/` — **skipped** if `server/node_modules/.bin/electron-builder` already exists
3. `node scripts/build-customer.js <channel>` (or `build-all.js`)
4. `node scripts/publish-update.js …` (only if BUILD &amp; PUBLISH was clicked)

The `dist:server` npm script was changed from `cd server && npm install && npm run dist` to `cd server && npm run dist` so that the second build never re-installs server deps redundantly — the admin chain (or `build-customer.bat` for CLI users) is the **single** source of `npm install`. A `dist:server:fresh` script preserves the install+build behavior for one-shot CLI invocations from a clean checkout.

This guarantees the previous `'vite' is not recognized` failure on a fresh RDP cannot recur, AND that re-clicking BUILD on a warm checkout goes straight to `vite build` with no install phase. The topbar shows a `DEPS: OK / PARTIAL / MISSING` pill driven by `/api/admin/status` and `/api/admin/customers` responses, with a yellow banner explaining what will happen on the next build.

### Granular Failure Reporting
`scripts/build-customer.js` wraps each major phase in a `substep('label', fn)` helper that emits machine-readable markers:
- `[SUBSTEP_BEGIN] vite build (launcher)`
- `[SUBSTEP_END_OK]` / `[SUBSTEP_END_FAIL]`

Phases tracked: `sync logo + rebrand source`, `vite build (launcher)`, `electron-builder (launcher)`, `electron-builder (server)`, `collect artifacts`. The admin server's `runStep` parses these from the child's stdout/stderr in real time and tracks the most recent unresolved substep on `job.currentSubstep`. When the chain step exits non-zero, `failedStep` is set to `<step.label> > <substep>`, so the operator's red top-of-console banner reads e.g. `Build customer "example-cafe" > electron-builder (launcher)` instead of the bare `Build customer ...`.

### Placeholder URL Protection
Customer `updateServer` URLs are classified by `classifyUpdateServerUrl(url, requestIp)` into one of three states:
- `placeholder` — URL matches `/YOUR[-_.]?(RDP|SERVER|IP|HOST)|change[-_]?me|example\.com|placeholder|configure[-_]?your/i`, or hostname is empty / URL invalid
- `loopback-when-remote` — hostname is `localhost` / `127.x` / `::1` AND the admin client is itself non-loopback. (Honors `X-Forwarded-For` only when the immediate connection is loopback, via `app.set('trust proxy', 'loopback')`, so remote clients can't spoof.)
- `''` — URL is fine

Both warning states surface as:
- A yellow border + inline warning block on the customer card (with copy specific to each cause)
- A `!! WARNING` block prepended to the build console BEFORE the chain runs

The job-end SSE event includes a `failedStep` field. Synthetic end events for clients reconnecting to a finished failed job include `failedStep` too. The admin UI shows a dedicated `#job-error-banner` (red gradient, glow + pulse animation) at the TOP of the BUILD CONSOLE on any non-zero exit, with the failed step + exit code surfaced large.

## OTA IPC Surface

Exposed on the renderer via `window.electronAPI.ota`:
- `getStatus()` → current channel/version/server, busy flags
- `checkNow()` → forces an immediate update check
- `restart()` → restart the app to apply staged update
- `on(event, handler)` → subscribe to:
  - `ota:update-available`
  - `ota:download-start`
  - `ota:download-progress`  (`{ downloaded, total, percent }`)
  - `ota:verifying`
  - `ota:applying`
  - `ota:ready-to-restart`
  - `ota:error`

The companion server uses the same IPC names but exposed via `window.serverAPI.ota`.

## Save & Load System

Client-server system for game save data persistence across diskless PCs.

### Server
- Standalone portable Electron app (target: `zip` + `dir`)
- Express REST API with SQLite database (sql.js)
- Starts hidden to system tray, auto-starts API on launch
- User auth with bcrypt + JWT tokens
- Game saves stored as zip archives per user
- Receives OTA updates independently from the launcher (channel `<base>-server`)

### Client (launcher)
- Save & Load button in title bar opens modal
- Login/Register flow with session auth
- Pure Node.js zlib for zip/unzip (no external deps)
- Env var expansion for game save paths

## OTA Hardening (Task #3)

The OTA updater (`electron/updater.js` and `server/electron/updater.js`) includes
several defensive features that operators should know about:

- **Orphan-exe cleanup with allowlist.** After an OTA stages a new build, the
  next launch sweeps any leftover `*.exe` from previous rebrands. To prevent
  this from deleting legitimate sibling tools (ffmpeg.exe, helper utilities,
  etc.), two allowlists are honored:
  - `manifest.keepExes` — published per-release by the update server.
  - `.ota-keep-exes.json` — operator-controlled file dropped next to the
    running exe. Three shapes accepted: a plain JSON array, `{"keep":[...]}`,
    or `{"keepExes":[...]}`. Entries are basename-normalized and
    case-insensitive.
- **Cleanup marker carries `nextExe`.** `.ota-cleanup.json` now records the
  exact basename of the new exe to hand off to. If an operator double-clicks
  the OLD exe after an OTA, the old process detects it's marked as an orphan
  and immediately spawns the recorded `nextExe` (no heuristic guessing about
  unrelated installers in the install dir).
- **Auto-restart drains in-flight requests.** server.exe quits ~7s after a
  successful stage, but the auto-quit waits (capped at 25s) for in-flight
  HTTP requests to finish first, so a 500MB save upload isn't sliced. The
  request counter is wired in via middleware in `server/electron/api.js`.
- **Idempotent helpers.** `scheduleAutoQuitAfterStage` and
  `scheduleSelfRelaunch` are both safe to call multiple times — the second
  call is a no-op with a log line.
- **Belt-and-braces relauncher.** On Windows, after the auto-quit, a detached
  `cmd.exe /c "ping … && start <exe>"` keeps the server coming back up even
  if the OS auto-start mechanism is misconfigured.
- **Per-exe identity sidecar (`.ota-current-exe.json`).** This is the
  PRIMARY signal an OLD exe uses to detect "I am no longer canonical".
  Written by every successful apply step at `<appRoot>/.ota-current-exe.json`
  with `{exe, version, written}`. Because it lives next to the exes (not
  inside the OTA-mutated asar) and is keyed by the canonical exe basename,
  an OLD exe relaunched after an OTA sees its own basename ≠ sidecar.exe
  and hands off — even when the asar's package.json has already been
  rewritten with the NEW version. Reading the sidecar is tier-1 in
  `pickSuccessorExe`; cleanup-marker `nextExe` is tier-2; `discoverNewExe`
  is gated to "no signal at all" (tier-3 fallback only).
- **Version-mismatch self-defense (degraded fallback).** When no sidecar
  is present yet (very first launch, pre-rollout state), init() falls
  back to comparing the bundled version (package.json) against the on-disk
  advertised version (ota-config.json). Acknowledged-imperfect path —
  the architect flagged that an OTA mutates package.json in place, so
  this tier alone would let an OLD exe see "bundled == advertised" and
  skip the handoff. The sidecar tier exists specifically to close that
  gap. Test hook for the degraded path: `OTA_TEST_BUNDLED_VERSION`.
- **Test coverage.** `node walok/electron/test-updater.js` runs 203 pure-Node
  tests covering all of the above. The earlier rebrand-flow tests
  (`tests/test-rebrand-update.js`, `tests/test-server-rebrand.js`) were
  removed at the user's request after a field bug shipped through them —
  unit tests that pass while production fails are worse than no tests.
